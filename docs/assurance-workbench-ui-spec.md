# Assurance Workbench UI Spec

## Purpose

Build a local web UI that helps users run `assurance-cli` evidence-gathering workflows without replacing the CLI.

The UI should make common assurance queries easier to configure, run, review and persist into a local Workbench folder. It should not introduce a central database, RAG system or GenAI dependency in the first version.

The CLI remains the stable evidence engine. The UI is a convenience layer.

## Goals

- Keep `assurance-cli` standalone and usable from terminal, VS Code, Copilot agents and scripts.
- Provide a form-based interface for common evidence pack runs.
- Save all evidence and run metadata into the user’s Workbench.
- Display gathered evidence in an organised, review-friendly way.
- Add deterministic, explainable analysis where possible.
- Avoid RAG and local LLM integration for the initial version.

## Non-Goals

- Do not replace the CLI.
- Do not store credentials in the UI.
- Do not create or mutate Confluence, Jira, Azure or Dataverse resources.
- Do not build a multi-user hosted service.
- Do not introduce vector databases, embeddings or RAG.
- Do not require an Ollama/local model dependency.

## Suggested Architecture

Use two separate projects:

```text
assurance-cli/
  assurance_cli/
  docs/

assurance-workbench-ui/
  backend/
  frontend/
```

The UI can depend on an installed `assurance` command or a configured path to the CLI virtualenv.

For the first version, prefer subprocess execution:

```bash
/path/to/assurance-cli/.venv/bin/assurance ...
```

This keeps the boundary honest and ensures the UI exercises the same behavior users get from the terminal.

Later, the CLI package can expose a cleaner Python API for library-mode integration if needed.

## Technology Choice

Recommended first version:

- Backend: FastAPI
- Frontend: simple React/Vite or HTMX
- Persistence: files in the Workbench
- Optional metadata index: local JSON files only

Avoid a database until file scanning becomes too slow or awkward.

## Workbench Persistence Model

The Workbench is the persistence layer and audit trail.

Suggested layout:

```text
Workbench/
  30 Workbenches/<Assurance Area>/
    02-Evidence/
      runs/
        2026-06-24-0915-booking-allocation/
          request.json
          command.txt
          evidence-pack.md
          stdout.log
          stderr.log
          analysis.json
          analysis.md
```

`request.json` should contain the selected UI options:

```json
{
  "topic": "booking allocation",
  "preset": "architecture",
  "sources": {
    "confluence": true,
    "jira": true,
    "azure": true,
    "dataverse": false
  },
  "confluence_space": "SPACE",
  "jira_project": "PROJ",
  "azure_resource_group": "rg-example-dev",
  "output": "evidence-pack.md",
  "created_at": "2026-06-24T09:15:00+01:00"
}
```

`command.txt` should contain the exact CLI command used.

`analysis.json` and `analysis.md` are deterministic analysis outputs generated by the UI, not GenAI output.

## Initial Screens

### 1. Run Evidence Pack

Primary screen for starting a new evidence run.

Inputs:

- Topic text input.
- Preset dropdown:
  - `none`
  - `architecture`
  - `dataverse`
  - `scaling`
- Source checkboxes:
  - Confluence
  - Jira
  - Azure
  - Dataverse
- Confluence space input.
- Jira project input.
- Azure resource group input.
- Output folder selector under the Workbench.
- Limit input.
- Include Jira comments checkbox.
- Refresh cache checkbox.
- No cache checkbox.

Actions:

- Preview command.
- Run.
- Cancel running process.
- Open output folder.
- Open result in VS Code.

### 2. Run Results

Displays the output from a completed run.

Sections:

- Run metadata.
- Command executed.
- Evidence pack Markdown preview.
- Source sections:
  - Confluence
  - Jira
  - Azure
  - Dataverse
- Gaps and warnings.
- Deterministic analysis.

The Markdown preview should preserve tables and headings.

### 3. Previous Runs

Lists previous runs by scanning the Workbench evidence folder.

Columns:

- Timestamp.
- Topic.
- Preset.
- Sources.
- Output path.
- Run status.

Actions:

- Open run.
- Open folder.
- Re-run using the same `request.json`.

### 4. Settings

Local-only configuration:

- Path to `assurance` executable.
- Default Workbench evidence root.
- Default Confluence space.
- Default Jira project.
- Default Azure resource group.
- Whether to use `--refresh` by default.

Do not store API tokens here.

## CLI Invocation

The UI should generate commands from structured options.

Example:

```bash
assurance report evidence-pack "booking allocation" \
  --preset architecture \
  --confluence-space SPACE \
  --jira-project PROJ \
  --include-azure \
  --azure-resource-group rg-example-dev \
  --limit 10 \
  --out "/path/to/Workbench/30 Workbenches/X/02-Evidence/runs/2026-06-24-0915-booking-allocation/evidence-pack.md"
```

If Confluence is unchecked:

```bash
--skip-confluence
```

If Jira is unchecked:

```bash
--skip-jira
```

If Azure is checked:

```bash
--include-azure
```

If Dataverse is checked:

```bash
--include-dataverse
```

## Deterministic Analysis

The first version should provide non-GenAI analysis only.

Analysis should be transparent, rule-based and repeatable.

Potential checks:

### Evidence Coverage

- Confluence evidence present.
- Jira evidence present.
- Azure evidence present when requested.
- Dataverse evidence present when requested.
- Evidence pack contains a gaps section.
- Commands completed without warnings.

### Freshness

- Confluence pages older than a configured threshold.
- Jira issues not updated recently.
- Evidence run older than a configured threshold.

### Azure Risk Flags

- Public network access appears enabled.
- Function app settings could not be retrieved.
- Managed identity appears missing.
- Role assignments evidence missing when Azure was requested.
- Resource group snapshot missing.

### Jira Risk Flags

- No Jira issues found.
- Many unresolved issues.
- High-priority issues present.
- Stale open issues.
- Missing assignee.

### Documentation Risk Flags

- No architecture page found.
- No operational readiness evidence found.
- No security evidence found.
- Confluence content was truncated.

Each analysis finding should include:

- Rule ID.
- Severity: `info`, `warning`, `concern`.
- Evidence source.
- Explanation.
- Suggested follow-up question.

Example:

```json
{
  "rule_id": "azure.function_settings_unavailable",
  "severity": "warning",
  "source": "Azure",
  "explanation": "Function app settings could not be retrieved for one or more apps.",
  "question": "Does the team have permission to verify app settings and secret references?"
}
```

## Security And Safety

- Never display `.env` contents.
- Never print API tokens.
- Treat raw JSON output as potentially sensitive.
- Prefer Markdown outputs with built-in redaction.
- Do not provide UI controls for mutating commands.
- Do not call `az`, `pac` or Atlassian APIs directly from the UI in v1; call `assurance-cli`.
- Clearly mark failed commands and partial evidence.

## Error Handling

Common errors should be actionable:

- CLI not found:
  - Show configured path.
  - Suggest installation command.
- Missing Atlassian env vars:
  - List missing variable names only.
  - Do not ask the user to paste tokens into chat.
- Azure not logged in:
  - Suggest `az login`.
- Power Platform CLI not found:
  - Suggest installing `pac` or configuring PATH.
- Command timed out:
  - Preserve partial logs.
  - Allow retry.

## Process Management

The backend should run CLI commands as child processes.

Requirements:

- Stream stdout/stderr to the UI.
- Capture stdout/stderr to files.
- Support cancellation.
- Record exit code.
- Use a timeout with a generous default.
- Avoid shell interpolation; pass argv arrays to subprocess calls.

## Future Options

Possible later additions:

- Library-mode integration with `assurance_cli` internals.
- Git integration to commit evidence packs into the Workbench.
- Local model analysis via Ollama/Qwen, explicitly marked as draft.
- Export to PDF.
- Comparison between two evidence runs.
- Configurable rule packs for deterministic analysis.

## Suggested First Slice

Build the smallest useful UI:

1. Settings page for `assurance` executable path and Workbench evidence root.
2. Evidence Pack form with topic, preset, source checkboxes and scope fields.
3. Command preview.
4. Run command and save output into a timestamped Workbench folder.
5. Render `evidence-pack.md`.
6. Generate a small deterministic `analysis.md` with coverage/gap checks.

This delivers value without changing `assurance-cli` or introducing AI/RAG complexity.
